ADR: 알림톡 비동기 발송 아키텍처

2026-04-02 | S-Class Backend

컨텍스트

Commission 생명주기(생성, 주제 제안, 자료 보강 요청, 티켓 처리 완료)에 걸쳐 카카오 알림톡(NCP Alimtalk v2)을 발송해야 한다.

비즈니스 로직 트랜잭션과 외부 API 호출(NCP)을 어떻게 결합할지 결정이 필요했다.

결정

✅ @Async + @TransactionalEventListener(AFTER_COMMIT)

@Async
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
fun handle(event: CommissionAssignedEvent) {
    val teacher = userAdaptor.findById(event.teacherUserId)
    val phoneNumber = teacher.phoneNumber ?: return
    commissionNotificationSender.sendCommissionAssigned(...)
}

UseCase에서 ApplicationEventPublisher.publishEvent()로 이벤트를 발행하면, 트랜잭션 커밋 후 별도 스레드에서 알림톡이 발송된다.

이유

AFTER_COMMIT 데이터 정합성

트랜잭션이 성공적으로 커밋된 후에만 알림 발송. DB 저장 실패/롤백 시 알림이 나가지 않는다.

예: 의뢰 저장이 실패했는데 "새 의뢰가 배정됐습니다" 알림이 나가면 안 됨.

@Async 응답 지연 없음

NCP API 호출 시간(~수백ms)이 HTTP 응답에 포함되지 않는다. 클라이언트는 DB 저장 완료 시점에 즉시 응답을 받는다.

관심사 분리

알림 발송 로직이 비즈니스 UseCase와 분리된다. 알림 실패가 트랜잭션에 영향을 주지 않는다.

트레이드오프

항목현황리스크
재시도 없음 NCP API 호출 실패 시 예외가 로그에만 기록됨 낮음 NCP 일시 장애 시 알림 유실
이벤트 유실 가능성 commit 직후 ~ async 실행 전 서버 크래시 시 발송 안 됨 낮음 실제로 발생 빈도 매우 낮음
비동기 예외 처리 async 스레드의 예외는 caller로 전파 안 됨 모니터링 로그/APM에서만 확인 가능
왜 이 리스크가 수용 가능한가?

알림톡은 보조 기능이다. 발송 실패가 비즈니스 데이터 무결성에 영향을 주지 않으며, 사용자가 앱에서 직접 상태를 확인할 수 있다.

언제 업그레이드해야 하는가

아래 요구사항이 생기면 Outbox 패턴 또는 메시지 큐(Kafka/RabbitMQ)로 전환을 고려한다:

  • 알림 발송 보장이 비즈니스 SLA에 포함될 때
  • 알림 발송 실패 이력을 DB에 저장하고 재시도해야 할 때
  • 알림 종류가 많아져 async 스레드풀 튜닝이 필요해질 때
  • 마이크로서비스로 분리가 필요해질 때
현재 규모에서의 결론: @Async + @TransactionalEventListener(AFTER_COMMIT) 조합은 구현 복잡도 없이 데이터 정합성과 응답 지연 최소화를 동시에 달성하는 적절한 선택이다. 서비스 초기 단계에서 Outbox 패턴 도입은 오버엔지니어링이다.
