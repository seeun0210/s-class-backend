수업을 '시작'하는 API는 POST일까 PATCH일까

2026. 04. 26 · 약 6분 · PR #289

처음 컨트롤러를 만들려고 앉았을 때 한참을 멍하니 있었다. start, complete, record — 수업의 라이프사이클을 다루는 세 API가 필요했는데, 어떤 HTTP 메서드를 쓸 건지가 안 정해졌다.

"RFC를 보면 답이 있겠지"라고 시작했는데, 읽을수록 깔끔한 답이 없었다. 결국 세 API의 메서드를 다르게 줬다. 그 결정 과정을 정리한다.

요구사항

POST /api/v1/lessons/{id}/start    — 수업 시작 (SCHEDULED → IN_PROGRESS)
POST /api/v1/lessons/{id}/complete — 수업 종료 (→ COMPLETED)
PUT  /api/v1/lessons/{id}/record   — 시작/종료 시각을 사후에 한 번에 기록

같은 도메인의 세 API인데 메서드가 둘로 갈렸다. 처음엔 "통일하는 게 깔끔하지 않나"라고 생각했고, 그래서 가장 먼저 떠오른 후보가 PATCH로 통일이었다.

첫 번째 후보: 셋 다 PATCH

가장 자연스럽게 떠오른 안이었다. 셋 다 기존 lesson의 일부 필드를 수정하는 동작처럼 보였으니까.

  • PATCH /lessons/{id} with {"action": "start"}
  • 또는 PATCH /lessons/{id}/status 같은 sub-resource

그런데 두 가지가 걸렸다.

1. URL이 동사로 끝나는 게 어색했다

PATCH /lessons/{id}/start. 이게 의미상 깔끔한가? PATCH는 리소스의 부분 수정인데, "start"는 리소스가 아니라 행위다. 명사 자리에 동사가 들어가는 셈이라 어딘가 어색했다.

2. 멱등성 의미가 충돌하는 줄 알았다 (← 잘못 알고 있었다)

처음엔 이렇게 생각했다. "두 번째 호출이 거절돼야 하는 비멱등 동작이니 PUT/PATCH랑 안 맞는다." 그래서 "POST를 써야 한다"고 결론냈다.

근데 RFC 9110을 다시 읽으면서 깨달았다. PUT의 멱등성은 "같은 요청을 반복해도 서버 상태가 같게 유지된다"는 것이지, 두 번째 호출이 거절되면 안 된다는 게 아니다. PUT으로 리소스 만들고 두 번째에 409 Conflict 반환하는 건 멱등성 위반이 아니다 — 거절돼서 상태가 변하지 않은 것이니까.

⚠️
잘못 알고 있던 것 ❌ "PUT/PATCH는 두 번 호출해도 둘 다 성공해야 한다"

✅ "PUT/PATCH는 두 번 호출 후의 서버 상태가 한 번 호출 후와 같아야 한다." 두 번째가 거절돼서 상태가 변하지 않으면 멱등성은 그대로 만족된다.

이걸 알고 나니 "POST를 써야 한다"는 강제 논리가 약해졌다. 더 본질적인 기준이 필요했다.

본질로 돌아가서 — 결정 기준 3개

RFC와 사례들을 다시 정리하면서 세 가지 질문을 stack으로 만들었다.

1. URL path가 명사인가, 동사인가?
   → 동사: POST + verb path (RPC-style action)
   → 명사: 다음 질문

2. 같은 body로 두 번 호출 시 최종 상태가 같은가?
   → 같음: PUT (전체 대체) 또는 PATCH (부분 수정)
   → 다름: POST

3. 단순 데이터 수정인가, 부수효과가 있는가?
   → 부수효과(상태 전이/외부 호출/이벤트 발행): POST가 의미 명확
   → 순수 데이터 수정: PUT / PATCH

이 stack으로 세 API를 다시 봤다.

각 API의 결정 — start / complete → POST

  • URL이 명백히 동사 (/start, /complete)
  • 의미상 명확히 행위임 — "수업을 시작한다", "수업을 종료한다"
  • 부수효과가 큼: status 전이 + timestamp 자동 기록 + (앞으로) Meet 링크 생성/알림 트리거 가능
💡
POST를 고른 진짜 이유 멱등성이 아니라 시맨틱이다. RPC-style 액션은 RFC 9110의 RESTful method 의미와 정확히 매핑되지 않는다. 행위는 POST + verb path가 가장 의도 전달이 명확하다 — Stripe(POST /v1/charges/{id}/capture), GitHub(POST /repos/.../merge), Google AIP-136(:cancel, :archive custom verb) 모두 같은 선택이다.

각 API의 결정 — record → PUT

이건 좀 다르다.

  • URL이 명사 (/record — "이 수업의 기록")
  • 같은 body로 두 번 호출 → 둘 다 같은 결과 (이미 기록돼 있으면 거절, 그 외엔 동일한 시각으로 채워짐)
  • 행위라기보다 데이터 채우기 — 오프라인 사후 기록 use case

PATCH도 후보였는데, PATCH는 "부분 수정"이고, record는 startedAt + completedAt을 한 번에 다 채우는 것이라 전체 대체 시맨틱의 PUT이 더 맞았다.

잠깐, PUT과 PATCH는 정확히 뭐가 다른가

이 결정을 하면서 PUT과 PATCH의 차이를 다시 정리했다. 막연히 "PUT은 전체, PATCH는 부분"으로 알고 있었는데, 멱등성까지 같이 보면 더 명확하다.

PUTPATCH
시맨틱전체 대체 (replace)부분 수정 (partial update)
body 의미리소스의 완전한 새 표현변경할 일부 또는 변경 명령
멱등성멱등 보장 (RFC 9110)구현에 따름 — 보장 안 됨
예시PUT /users/1 로 전체 user 객체 교체PATCH /users/1 로 {"email": "new"}만 수정

이 기준으로 보면 record는 명확히 PUT이다 — 시작/종료 시각을 둘 다 한꺼번에 채우니까 "기록(record)이라는 sub-resource의 완전한 새 표현"이다. 만약 startedAt만 따로 수정하는 API였다면 PATCH가 더 맞았을 거다.

✅
record가 PUT인 이유 sub-resource(시간 기록)를 전체적으로 채우는 동작. 멱등성 충족(RFC 9110 § 9.2.2), URL은 명사, 부수효과 적음. GitHub의 PUT /repos/.../issues/{n}/lock이 가장 가까운 사례다 — "lock"이라는 동사를 PUT으로 쓰지만, 두 번 lock 해도 결과 같음(이미 locked → 그대로). 멱등성이 만족되니 PUT으로도 정당.

비교: 다른 회사들은 어떻게 했나

조사하면서 흥미로웠던 사례들이다.

회사패턴예시
Stripe행위는 POST + verbPOST /charges/{id}/capture
GitHub멱등 행위는 PUTPUT /issues/{n}/lock
Google AIP-136RPC 액션은 POST + custom verb:cancel, :archive

일관된 정답은 없다. 회사마다 자기 도메인에 맞춰 결정한다. 우리도 그래야 했다.

트레이드오프 — 일관성을 포기한 대가

한 PR 안에서 POST와 PUT을 섞은 건 비용이 있다.

단점받아들인 이유
일관성 깨짐 — 셋 다 POST면 더 단순시맨틱 명확성이 일관성보다 우선이라 판단
FE 개발자 학습 부담API 문서에 메서드 선택 근거 명시로 보완
팀 컨벤션 만들기 어려움결정 stack을 컨벤션으로 문서화 (이 글이 그 시작)
⚠️
일관성 vs 명확성 둘 다 좋은 가치인데 충돌할 때가 있다. 이번엔 명확성을 골랐다. 다른 팀은 다르게 고를 수 있고, 그게 틀린 게 아니다 — 트레이드오프를 의식하고 골랐는지가 중요하다.

다시 처음으로 — 깨달은 것

처음엔 "RFC가 결정해주겠지"라고 시작했는데, RFC는 결정해주지 않는다. 그건 최소한의 시맨틱만 정의하고, 그 위에 어떻게 매핑할지는 팀의 컨벤션 + 도메인 의미 영역이다.

💡
이번에 배운 것 1. PUT의 멱등성은 "결과 상태 동일"이지 "두 번 다 성공"이 아니다.

2. RPC-style 행위는 RESTful method와 깔끔하게 매핑되지 않는다 — POST + verb path가 가장 의도 전달이 명확.

3. 일관성과 의미 명확성 사이에서 후자를 골랐다. 트레이드오프를 의식하고 골랐다는 게 핵심이다.

결정 stack을 컨벤션으로

다음 endpoint를 만들 때는 이 stack대로 묻는다.

URL path가 동사?           → POST + verb path
URL path가 명사 + 부수효과 큼  → POST
URL path가 명사 + 멱등          → PUT (전체) / PATCH (부분)
순수 데이터 수정              → PUT / PATCH

이 글이 우리 팀의 최소 컨벤션의 시작이 되길 바라며, 이번 PR이 첫 적용 사례다.

✅
한 줄 요약 같은 도메인 안의 세 API에 POST와 PUT을 섞었다. 일관성보다 시맨틱 명확성을 우선했고, 그 결정 근거를 글로 남겨 다음 사람이 같은 고민을 반복하지 않게 했다.
